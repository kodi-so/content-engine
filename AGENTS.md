# Agent Instructions

- This React app uses Tailwind utilities for UI styling. Use Tailwind classes in components for new layout and presentation work.
- Do not add new standalone CSS files or feature-specific CSS blocks unless the user explicitly asks for stylesheet changes.
- Reuse existing shared components and styling primitives before introducing new surface area.
- Use the shared custom dropdown/select primitive for all dropdown UI. Do not add native `<select>` elements for product-facing controls.
- For UI work, do a brief design pass before implementation: identify the user's task, choose the right product pattern, and avoid surfacing backend concepts as generic panels.
- Prioritize calm, beautiful, task-appropriate product UI. Do not default to card-heavy dashboards for settings, admin, or operational workflows; use quieter sections, dividers, tables, inspectors, or forms when they fit better.
- Treat visual quality as part of done. For visual review, rely on screenshots provided by the user instead of using Codex, Browser, or other visual inspection tooling. Use those screenshots to identify and fix awkward hierarchy, cramped controls, excessive boxes, unclear copy, or incoherent spacing before finalizing.
- Keep maintainability in view during every feature, fix, and refactor. When a file starts mixing page orchestration, data shaping, event-heavy logic, and large UI sections, extract focused components, hooks, helpers, or feature modules before it becomes a broad cleanup project.
- For chatbot or agent behavior, do not implement semantic understanding, conversational memory, or user intent with hard-coded phrase lists, regex intent detectors, or special-case prompt rewrites. Treat prior messages and tool results as conversation context and use the agent/model to reason over that context. Keep runtime normalization limited to mechanical constraints such as provider-required parameter bounds, schema cleanup, and deterministic safety checks.
- Agent architecture note, grounded in current model docs: preserve multi-turn state by passing prior user/assistant/tool messages into the model, or by using a session/conversation-state mechanism; let the model decide whether to answer or call tools from tool descriptions and the current conversation. Do not replace conversation state with phrase-triggered routing. See OpenAI Conversation State docs, OpenAI Agents SDK Sessions docs, and Anthropic/Claude tool-use docs for this pattern.
- Prefer domain-oriented subdirectories such as `src/features/<area>/`, `src/lib/<domain>/`, and focused Convex helper modules for reusable logic. Keep top-level pages and Convex public function files easy to scan; move implementation detail behind named modules when the boundary is clear.
- Do not split files only to satisfy a line count. Split when the extracted piece has a cohesive responsibility, reduces duplication, makes tests or future changes easier, or matches an existing local pattern.
- Before finishing substantial work, do a quick maintainability pass: check the largest touched files, look for duplicated logic introduced by the change, confirm imports still follow sensible domain boundaries, and run the relevant validation command.
